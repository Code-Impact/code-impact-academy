Week 10 · GitOps & cloud

GitOps with Actions

Your repo starts working for you. You add Continuous Integration (checks on every push) and Continuous Deployment (auto-deploy the front end), and learn the GitOps mindset: git is the source of truth, and a push — not a human — is what triggers the pipeline.

≈ 3 hours Phase: GitOps & cloud Ship: push → automatic build and deploy

🎯 By the end of this week

  • A GitHub Actions workflow runs on every push (lint / test / build).
  • Failing checks block a broken merge; green checks give you confidence.
  • The front end auto-deploys on push to main.
  • You’ve chosen which checks are required, and protected main.
  • You can read a workflow run and explain what each step did.
🔁

GitOps in one line: the repo is the plan, and the pipeline makes reality match it. No more deploying by hand or "it works on my laptop". You push; automation checks and ships. Git history becomes the record of what happened.

1

Understand GitOps

≈ 15 min · read

Three ideas carry the whole week:

  • Git is the source of truth — the code in main is what should be running.
  • A push is the trigger — changes flow through automation, not manual steps.
  • Checks gate changes — code that fails the pipeline doesn’t reach main.
2

Add continuous integration

≈ 45 min

The starter ships .github/workflows/ci.yml — adapt it to your front end. A minimal pipeline checks out the code, installs deps, and runs lint / test / build.

# .github/workflows/ci.yml (sketch)
name: CI
on: [push, pull_request]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci
      - run: npm run lint
      - run: npm test --if-present
      - run: npm run build
Decide Pipeline stages & which checks must pass

What runs on every push, and which checks are required to merge? Lint + build is a sensible minimum; add tests as you write them.

Your call: your stages and your "must be green to merge" set.

3

Make it fail (on purpose)

≈ 20 min

A pipeline you’ve never seen fail is a pipeline you don’t trust. Break something small — a lint error or a failing assertion — push it, and watch the run go red.

Open the failed run, read the log, find the step that failed, fix it, and watch it go green. Reading CI logs is the actual skill here.

4

Confirm continuous deployment

≈ 20 min

You already have CD from week 4: Vercel deploys on push. Make it deliberate:

  • Pushing to main updates production.
  • Opening a pull request gives you a preview deployment — a live URL for that branch.

Now every change is visible before it ships. Use PRs and preview URLs to review UI changes like a real team.

5

Protect main

≈ 20 min

Tie it together with a branch protection rule on main: require your CI check to pass before a PR can merge.

From here on you work on branches, open PRs, let CI vote, and merge on green. That’s the professional loop — and it’s enforced, not just intended.

✅

Ship: a push now triggers automatic checks and an automatic deploy — no manual steps in the path to production.

✅ Definition of done

  • ✓ A GitHub Actions workflow runs lint/test/build on every push and PR.
  • ✓ You’ve watched a run fail and fixed it from the logs.
  • ✓ Front end auto-deploys to production on main; PRs get preview URLs.
  • ✓ main is protected — required checks must pass to merge.
  • ✓ You can open a workflow run and explain each step.

🎤 Friday demo — 5 minutes

  1. Open a PR and show CI running on it.
  2. Show a red run and the log line that caused it.
  3. Merge on green and show the automatic production deploy.
  4. Show the preview URL from a PR.
  5. Which check you made required, and why.

Stuck? Common snags

My workflow doesn’t run at all

Check the file is at .github/workflows/*.yml, valid YAML, and that the on: triggers match your branch. A wrong indent or branch filter silently disables it — the Actions tab shows parse errors.

How do I use secrets in Actions?

Store them as repository or environment secrets in GitHub settings and reference them via the secrets context in the workflow. Never echo them to logs or commit them.

CI is slow

Cache dependencies (actions/setup-node has a cache option), run independent jobs in parallel, and only build what changed. A tight pipeline is one people actually wait for.

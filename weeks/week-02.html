Week 02 · Set up & spec

Write the spec

Before a single line of code, you decide what you're building and why. This week you learn spec-driven development and turn a fuzzy idea — "a fitness tracker" — into a clear, honest SPEC.md: who it's for, what it does in v1, what it deliberately won't do, and how you'll know it's done.

≈ 2–3 hours Phase: Set up & spec Ship: SPEC.md v1

🎯 By the end of this week

  • You can explain what spec-driven development is and why the spec comes before the code.
  • Your repo has a filled-in docs/SPEC.md: summary, users, features, non-goals, data, sign-in, journeys, success.
  • You've made the key product call — which features are in v1, and which are deliberately out.
  • Your AI agent has reviewed the spec for gaps and contradictions, and you've acted on its notes.
  • SPEC.md v1 is committed and pushed.
🧭

What a spec is — and isn't. A spec answers what and why in plain words. It is not code, not database tables, not API routes — that's next week's ARCH.md. If you catch yourself writing CREATE TABLE, stop: you've jumped ahead.

1

The spec-driven mindset

≈ 15 min · read

Most bugs aren't typos — they're misunderstandings about what was supposed to happen. Spec-driven development front-loads that thinking. You write down the plan first, so that you (and your AI agent) build the right thing, not just a thing.

  • The spec is the source of truth. When code and spec disagree, one of them is wrong — and you decide which, on purpose.
  • Small and honest beats big and vague. Three features you can demo are worth more than ten you can't.
  • The agent works from your words. A sharp spec produces sharp code; a hand-wavy spec produces a hand-wavy mess.
🧠

RememberYou may let AI write the code — but you may not ship what you can't explain. The spec is where your understanding starts.

2

Open the template

≈ 5 min

The starter kit already ships a SPEC.md template full of guided prompts. Open docs/SPEC.md in your repo (from Week 1). Prefer to skim it first? Read the template on GitHub.

You'll fill in these sections, top to bottom:

1. One-line summary        # what & who, in one sentence
2. The people (users)      # who uses it, what they want
3. Core features (v1)      # the 3–6 that matter
4. Non-goals               # what you will NOT build yet
5. The data                # the "things" the app stores
6. Sign-in                 # public vs private
7. Key user journeys       # 2–3 flows, start to finish
8. Success criteria        # how you know v1 is done
9. Open questions & risks  # what you're unsure about

Delete each italic prompt as you replace it with your own answer. The thinking is the point — don't let the agent write this part for you yet.

3

Summary, users & the data

≈ 25 min

One-line summary

Force yourself into a single sentence. Example: "A private web app where I log workouts and see whether I'm hitting my weekly goals." If you can't say it in one line, you don't know it yet.

The people

Even a personal tracker has a user — say it clearly. Name the primary user, what they want to achieve, and what would make them stop using it (this last one quietly reveals your real requirements).

The data

List the main "things" the app remembers and their key fields — but don't design tables. That's architecture. For example:

# Thing → fields
Workout   → date, type, duration, notes
Goal      → metric, target, period
4

Features & non-goals

≈ 25 min

This is the heart of the spec — and the decision you'll defend on Friday.

Core features (v1)

Aim for 3–6 features. The test: if you can't demo it, it's not a feature yet. "Nice dashboard" isn't a feature; "see this week's total minutes" is.

Non-goals

Just as important — list what you're deliberately not building in v1 (social sharing, a mobile app, AI coaching, teams…). Non-goals are how you protect yourself from scope creep and actually ship.

Decide What's in v1 — and what's out

Everyone builds "a fitness tracker with Gmail login", but your v1 line is yours to draw. Pick the smallest set that's genuinely useful and demoable in 12 weeks.

Your call: the 3–6 features that make the cut, and an honest non-goals list. Write a sentence on why each borderline feature is in or out.

5

Sign-in, journeys & success

≈ 20 min

Sign-in

This app uses Google (Gmail) login via Supabase (you'll build it in week 5). For now just answer two questions: what can a signed-out visitor see, and what is private to the signed-in user?

Key user journeys

Walk through 2–3 things a user actually does, start to finish, in plain steps. For example:

Log a workout: sign in → tap "Add" → pick type & duration
              → save → see it in today's list

Success criteria

Write checks you can literally tick off — "I can log a workout and see it after refresh", "I only see my own data". These become your definition of "done" for the whole project.

6

Have your agent review it

≈ 15 min

Now use the agent — not to write the spec, but to pressure-test it. This is the loop from week 1: prompt → review → act.

Prompt to your agent:
"Review docs/SPEC.md as a skeptical engineer. Point out gaps,
contradictions, and any feature that isn't clearly demoable.
Do NOT change the file — just list your findings."

Read its findings critically — the agent is a reviewer, not a boss. Where it's right, you edit the spec. Where it's wrong, note why. Then commit:

git add docs/SPEC.md
git commit -m "SPEC.md v1: users, features, non-goals, data"
git push
✅

Spec changes first, always. For the rest of the course, when the plan changes you update SPEC.md before the code. It stays the source of truth.

✅ Definition of done

  • ✓ Every section of docs/SPEC.md is filled in — no italic prompts left.
  • ✓ You have 3–6 demoable v1 features and an explicit non-goals list.
  • ✓ Data "things", sign-in rules, 2–3 journeys, and tick-off success criteria are written.
  • ✓ The agent reviewed the spec and you acted on (or consciously rejected) its notes.
  • ✓ SPEC.md v1 is committed and pushed.
  • ✓ You can explain, out loud, why each feature is in or out of v1.

🎤 Friday demo — 5 minutes

  1. Your one-line summary and who the app is for.
  2. Your v1 features — and one thing you deliberately cut, and why.
  3. Walk one user journey end to end from the spec.
  4. One gap the agent's review caught that you'd missed.
  5. What architecture question you're most unsure about for next week.

Stuck? Common snags

My feature list keeps growing

That's scope creep — the exact thing non-goals exist to stop. Move anything you're unsure about into Non-goals with a note "maybe v2". You can always promote it later; you can't un-ship a bloated v1.

I keep writing about tables and screens

You've drifted into architecture. Tables, columns and API routes belong in next week's ARCH.md. In the spec, say "the app stores workouts" — not "a workouts table with a uuid primary key".

Should I let the agent write the spec for me?

No — write it yourself this week. The agent is great at reviewing your spec for holes, but the product thinking is the skill you're here to build. If you outsource it, you'll be unable to explain your own app.

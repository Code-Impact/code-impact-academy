Week 04 · Build with AI

Front end, live on Vercel

Docs become a real, running thing this week. You'll direct AI agents to scaffold the React front end straight from your ARCH.md, run it on your machine, then deploy the first version to Vercel — a live URL you can send to a friend. No real login or data yet; that's weeks 5–6. This week is about a screen you can see.

≈ 3–4 hours Phase: Build with AI Ship: front end online on Vercel

🎯 By the end of this week

  • A running React app in your repo's frontend/, scaffolded with the agent from your ARCH.md.
  • It runs locally — npm run dev opens your screens in the browser.
  • Your chosen screens and component structure are in place (with placeholder/mock data).
  • The front end is deployed and live on Vercel from your GitHub repo.
  • You can explain what each component does and where the data will come from later.
🎨

Design first, generate second. The agent builds fast — so if you don't decide your screens, you'll get its idea of a fitness tracker, not yours. Spend the first 20 minutes on paper before you prompt.

1

Decide your screens

≈ 20 min

Pull up your spec's user journeys and turn them into a short list of screens and the components each needs. Keep v1 small — you can always add more.

# A typical v1
Screens:     Dashboard · Add workout · History
Components:  NavBar · WorkoutForm · WorkoutList
             · WorkoutCard · WeekSummary
Decide Screens & component structure

Which screens exist in v1, how you navigate between them, and how you break the UI into components. There's no single right answer — a flat three-screen app is a fine call.

Your call: the screen list and component tree. Jot it in ARCH.md so the doc stays the source of truth.

2

Scaffold with the agent

≈ 45 min

Now direct the agent to build it — pointing at your architecture, not a blank prompt. Work in the frontend/ folder from the starter kit.

Prompt to your agent:
"In frontend/, scaffold a React + Vite app (JavaScript, React Router).
Create the screens and components listed in docs/ARCH.md. Use mock
in-memory data for now — no backend or auth yet. Keep components small
and in their own files."

Build in small steps. Ask for one screen, review the diff, run it, then ask for the next. A giant "build the whole app" prompt gives you a giant diff you can't review — the opposite of the point.

🧠

The one ruleThe agent will happily generate a hundred lines you didn't read. Don't accept a component you couldn't explain in your demo — that's the next thing you learn, not the next thing you merge.

3

Run it locally

≈ 20 min

Install dependencies and start the dev server from inside frontend/:

cd frontend
npm install
npm run dev        # opens http://localhost:5173

Click through your screens. Broken? That's normal — feed the exact error back to the agent ("clicking Add throws this: …"), review its fix, and re-run. This is the build loop in miniature.

✅

Commit once it runs. A working local app is a real milestone: git add -A && git commit -m "Front end scaffold: screens + mock data" && git push.

4

Deploy to Vercel

≈ 30 min

Time to put it online. Vercel builds your app from GitHub and gives it a public URL — and later, every push will redeploy it automatically.

  1. Sign in at vercel.com with your GitHub account.
  2. Add New → Project, and import your repo.
  3. Set the Root Directory to frontend (important — your app lives in a subfolder). Vercel auto-detects Vite; leave the build command as detected.
  4. Click Deploy and wait for the build to finish.
⚠️

Build failed? The most common cause is the wrong Root Directory — it must point at the folder with package.json (frontend). Read Vercel's build log; it names the failing step. Fix locally, push, and it redeploys.

5

Verify & iterate

≈ 15 min

Open your *.vercel.app URL on your phone and on your laptop. It's really on the internet now. Make one small visible change (a title, a color), commit and push, and watch Vercel redeploy automatically — your first taste of the GitOps loop you'll formalise in week 10.

🔒

No secrets needed yet. With mock data there's nothing sensitive to configure. When Supabase arrives in week 5, its public keys go in Vercel's Environment Variables — never hard-coded in the repo.

✅ Definition of done

  • ✓ frontend/ holds a React app that runs with npm run dev.
  • ✓ Your v1 screens exist and match the component structure you chose.
  • ✓ The app is live on a Vercel URL, built from your GitHub repo.
  • ✓ A push to main triggers an automatic Vercel redeploy.
  • ✓ All changes are committed and pushed; no build errors.
  • ✓ You can walk through every screen and explain each component.

🎤 Friday demo — 5 minutes

  1. Open your live Vercel URL — show it works on a real device.
  2. Click through your screens and explain your component structure.
  3. Show one diff the agent produced and what you changed about it.
  4. Push a small change live and show the auto-redeploy.
  5. One thing that broke, and how you fixed it.

Stuck? Common snags

Vercel build fails but it runs locally

Almost always the Root Directory (set it to frontend) or a dependency that's installed globally on your machine but not in package.json. Read the build log, reproduce with a clean npm install locally, and let the agent fix the missing piece.

The agent built way more than I asked for

Reject the diff and prompt smaller: "just the Dashboard screen and its NavBar, nothing else". One screen per prompt keeps every diff reviewable — which is the whole skill this week.

Should I add login and a database now?

No — that's weeks 5 and 6. This week uses mock in-memory data so you can focus on the UI and getting a real deployment working. Wiring auth into a front end that already deploys cleanly is far easier than doing both at once.

My routes 404 on Vercel after refresh

That's a classic single-page-app rewrite issue. Ask the agent to add the standard Vite/SPA rewrite so all paths serve index.html, then let React Router handle routing. Commit, push, and Vercel redeploys.

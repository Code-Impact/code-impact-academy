Week 01 · Set up & spec

Liftoff: your AI cockpit

This week you don't build the app yet — you build the cockpit you'll fly for the next 12 weeks. Get Antigravity running, sign in, wire up GitHub, push an empty repo, and run your very first task through an AI agent. When week 2 starts, you should be thinking about your product — not fighting installers.

≈ 2–3 hours Phase: Set up & spec Ship: repo pushed + Antigravity running

🎯 By the end of this week

  • Antigravity is installed and open, signed in with your Gmail.
  • Git works on your machine and your identity is configured.
  • Your project repository lives on GitHub, with the starter kit and a first commit pushed.
  • You've run one small task through an AI agent and reviewed its diff before accepting it.
  • You've made two design calls — repo layout and agent autonomy — and written them down.
💡

New to any of this? That's expected. The point of week 1 is to get comfortable directing tools, not to memorise commands. Copy the commands, read what they do, and ask your agent to explain anything that's unclear.

1

Install Antigravity

≈ 15 min

Antigravity is an AI-first IDE — a fork of VS Code with an AI agent built into the editor. It's a full application you install, not a browser plugin. This is where you'll write specs, direct the agent, review diffs, and run your app all week.

  1. Go to antigravity.google and download the build for your operating system (Windows, macOS or Linux).
  2. Install it the normal way for your OS, then launch it.
  3. On first launch it offers to import your VS Code settings, themes and keybindings — accept if you already use VS Code, so it feels familiar from minute one.
⚠️

macOS blocked the app? If you see "app can't be opened because it is from an unidentified developer", open System Settings → Privacy & Security, scroll down, and click Open Anyway. This is normal for freshly downloaded apps.

2

Sign in with Gmail

≈ 5 min

Use a personal Gmail account and stick with it for the whole course. You'll reuse this same Google identity later for Supabase Auth and the "Sign in with Gmail" feature in your app, so keeping one account avoids confusion in week 5.

  1. In Antigravity, open the account / sign-in panel and choose Sign in with Google.
  2. Pick your personal Gmail, review what's requested, and approve.
  3. Confirm you're signed in — your account should show in the editor, and the AI agent panel should be available.
🔑

Write down which Gmail you used. Every account this course touches — GitHub, Supabase, Vercel, Google Cloud — should be reachable from this one inbox. One identity, fewer headaches.

3

Connect GitHub

≈ 15 min

GitHub is where your code lives and, later, where a push will automatically build and deploy your app. First make sure Git is installed locally, then connect your GitHub account.

Check Git is installed

Open a terminal (there's one built into Antigravity: View → Terminal) and run:

# should print a version number
git --version

If it's missing: on macOS run xcode-select --install; on Windows install Git for Windows; on Linux use your package manager (e.g. sudo apt install git).

Tell Git who you are

These two lines stamp your name and email onto every commit you make:

git config --global user.name "Your Name"
git config --global user.email "you@gmail.com"

Sign in to GitHub

  • Create a free account at github.com if you don't have one (use the same email where you can).
  • In Antigravity, sign in to GitHub through the built-in GitHub / Source Control panel and authorise it in the browser.
  • This lets the editor create repos and push on your behalf without pasting tokens by hand.
✅

Quick sanity check: if the Source Control panel shows your GitHub username and offers to "Publish to GitHub", you're connected.

4

Create your project repo

≈ 20 min

Now make the home for your project. You'll start from the starter kit — a scaffold with folders for the front end, back end and docs already in place — so week 2 begins with structure, not a blank page.

Get the starter kit

Download it from the course home page: Grab the starter kit → Starter repo (.zip). Unzip it somewhere sensible, like ~/projects/fitness-tracker, and open that folder in Antigravity (File → Open Folder).

Take a minute to look around. You'll see:

.
├── docs/
│   ├── SPEC.md          # WHAT you're building (week 2)
│   └── ARCH.md          # HOW it's built (week 3)
├── frontend/            # React app → Vercel
├── backend/             # API + Dockerfile → AWS
├── .github/workflows/   # GitOps: build/deploy on push
├── .env.example         # copy to .env, never commit .env
└── .gitignore

Publish it to GitHub

The easiest path is Antigravity's "Publish to GitHub" button in the Source Control panel — pick a name, choose Private, and it creates the repo and pushes for you. Prefer the command line? From inside the project folder:

# start tracking, make your first commit
git init
git add .
git commit -m "Initial commit: project scaffold"

# create the repo on GitHub and push (needs the GitHub CLI)
gh repo create fitness-tracker --private --source=. --push

# …or, if you made the empty repo on github.com first:
git remote add origin git@github.com:your-user/fitness-tracker.git
git branch -M main
git push -u origin main
🔒

Make it private for now, and never commit a real .env. The .gitignore in the starter already excludes it — that's deliberate. Secrets live outside git for the whole course.

Refresh your repo on github.com. Seeing the starter files there means the "ship" for this week is done.

5

Meet your agent: the loop

≈ 25 min

Here's the habit the whole course is built on. You don't type the code — you direct an agent and stay responsible for the result. Every task follows the same loop:

  1. Prompt — tell the agent, in plain words, what you want and why.
  2. Review the diff — read exactly what it proposes to change before accepting.
  3. Run it — actually execute the result and see it work (or not).
  4. Fix it — feed back what's wrong, or tweak it yourself, and go again.

Try it now — a tiny first task

Open the agent panel and give it something small and safe, so the point is the workflow, not the output. For example:

Prompt to your agent:
"Open README.md. Add a short section titled 'About me' with my
name and one sentence about the app I want to build. Keep the rest
of the file unchanged."

When it responds, do not just click accept. Read the diff: green lines are additions, red are removals. Ask yourself — did it touch only what I asked for? Then accept, and commit the change:

git add README.md
git commit -m "Add About me to README"
git push
🧠

The one ruleYou may let AI write the code — but you may not ship what you can't explain. If you couldn't describe what that diff did, that's the next thing you learn, not the next thing you merge.

Your design decisions this week

Two small calls are yours to make. There's no single right answer — decide, and jot a line about why so you can defend it at the Friday demo.

Decide Repo layout

The starter uses a monorepo: front end, back end and docs in one repository. That's the recommended default for this course — one place, one history, simpler deploys. The alternative is separate repos per part.

Your call: keep the monorepo (recommended), or split it — and note the reason.

Decide Agent autonomy

How much rope do you give the agent? At one end, it edits freely and you review afterward; at the other, you approve every single change first.

Your call for week 1: start conservative — review every diff before accepting. You can loosen this later once you trust it on a given kind of task.

✍️

Write it down. Ask your agent: "Add a short 'Decisions' section to the README noting my repo layout and how I'll review the agent's work." Review the diff, accept, commit. That's the loop again — and now your choices are on the record.

✅ Definition of done

You've finished week 1 when you can tick every box:

  • ✓ Antigravity opens and you're signed in with your Gmail.
  • ✓ git --version works and your user.name / user.email are set.
  • ✓ A private repo exists on GitHub with the starter kit and your first commit pushed.
  • ✓ You ran at least one agent task and reviewed the diff before accepting.
  • ✓ Your repo layout and autonomy decisions are written in the README.
  • ✓ You can open your repo on github.com and explain what each top-level folder is for.

🎤 Friday demo — 5 minutes

Same format every week. For week 1, walk us through:

  1. Antigravity open, signed in — show the agent panel.
  2. Your repo on github.com with the starter files.
  3. The diff from your agent task — explain what it changed and why.
  4. One thing that surprised you or went wrong during setup.
  5. What you plan to spec next week.
🗣️

Show the running tools and the diff — talk us through it, don't read slides. Owning "here's what confused me" is worth more than a flawless run.

Stuck? Common snags

Antigravity won't install or open

On macOS, use System Settings → Privacy & Security → Open Anyway after the first blocked launch. On Windows, right-click the installer and Run as administrator if it stalls. Make sure you downloaded the build that matches your OS and chip (Apple Silicon vs Intel).

git is "command not found"

Git isn't installed or your terminal hasn't picked it up. Install it (macOS: xcode-select --install; Windows: Git for Windows; Linux: your package manager), then close and reopen the terminal so it re-reads your PATH.

My push was rejected / it asks for a password

GitHub no longer accepts your account password over HTTPS. Easiest fix: sign in to GitHub inside Antigravity and use "Publish to GitHub", or install the GitHub CLI and run gh auth login once. After that, pushes just work.

The agent changed more than I asked for

Good — you caught it, which is the whole skill. Don't accept the diff. Reject it and re-prompt more tightly: name the file, say "change only this", and "leave everything else unchanged". Smaller, specific prompts give smaller, reviewable diffs.

I accidentally committed a secret / .env

Remove it from tracking with git rm --cached .env, confirm .env is listed in .gitignore, then commit. If it was a real key, rotate (regenerate) that key — anything pushed to GitHub should be treated as exposed.

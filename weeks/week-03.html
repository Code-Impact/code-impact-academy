Week 03 · Set up & spec

Spec → architecture

Your spec says what; now you decide how. This week you turn SPEC.md into ARCH.md: the big picture, a diagram, your components, a real data model, access rules, an API contract, and a deployment plan. No two students' architectures will be identical — and that's the point.

≈ 3 hours Phase: Set up & spec Ship: ARCH.md v1 + data-model sketch

🎯 By the end of this week

  • A filled-in docs/ARCH.md that a stranger could read to understand how your app fits together.
  • A diagram (boxes and arrows is fine) showing front end, auth, data and backend.
  • A data model — your spec's "things" turned into tables, columns and relationships.
  • Access rules in plain words, and a small API contract for your backend.
  • At least one decision & trade-off written down, ready to defend.
  • ARCH.md v1 committed and pushed.
🏗️

Architecture is where your spec meets reality. It's still a document, not code — but now you make concrete choices: where each piece runs, who owns which data, and what the backend is even for. Get this roughly right and week 4 onward is mostly execution.

1

Open the template & the big picture

≈ 20 min

Open docs/ARCH.md from the starter kit (or read it on GitHub). Start with section 1: describe, in 3–4 sentences, how the parts fit together — what talks to what.

The course stack gives you a well-worn shape to start from:

  • Front end (React) → runs on Vercel, talks to Supabase directly for simple things.
  • Auth + Database → Supabase (Gmail login + Postgres).
  • Backend API → your choice of language, in a Docker container, local first then AWS — for logic the front end shouldn't own.
2

Draw the diagram

≈ 20 min

A picture beats a paragraph. A simple boxes-and-arrows sketch is perfect — paste an image, or use a text diagram right in the doc:

[ Browser ]
     |
     v
[ Front end: React ] ----> [ Supabase: Auth + Postgres ]
   (Vercel)                        ^
     |                             |
     v                             |
[ Backend API: your choice ] -----+
   (Docker -> AWS)

Make it match your decisions. If your v1 has no backend logic yet, say so — an honest "the front end talks only to Supabase for now" is a valid architecture.

3

Components & the data model

≈ 35 min

Components

List each part, what it's responsible for, the tech, and where it runs. Keeping responsibilities clear now prevents the "why is this logic in the browser?" mess later.

Data model

Turn SPEC section 5 (your "things") into real tables — columns, types, and how they relate. Note which user owns each row; you'll need that for security next.

# Table: workouts
id         uuid   primary key
user_id    uuid   owner (from auth)
type       text
duration   int    minutes
performed  date

# one user has many workouts
4

Access rules & API contract

≈ 25 min

Access rules (security)

State, in words, who can read and write what. With Supabase you'll enforce this later with Row-Level Security — but write the rule in English first:

Rule: a user can only read and write rows where
      user_id = their own authenticated id.

API contract

If you have a backend, list its endpoints — method, path, what it needs, what it returns. Keep it small; you can add more later.

GET  /health         →  { "status": "ok" }
GET  /summary/week   →  weekly totals & current streak
                        (user comes from the auth token)
5

Secrets & deployment plan

≈ 15 min

Secrets & configuration

Name the secrets that will exist (Supabase URL + keys, backend port) and where they live — .env locally (git-ignored), platform settings in Vercel and AWS. Never in git.

Deployment plan

Match the course pipeline: what runs locally early, and what moves to the cloud at the end.

Front end   local dev server   →  Vercel
Data & auth  Supabase cloud     →  Supabase cloud
Backend     Docker on laptop   →  Docker on AWS (ECR + runtime)
6

Decide, then have the agent check it

≈ 25 min
Decide How the pieces fit & where logic lives

Your biggest architecture call: what belongs on the server vs the browser? A streak calculation could live in either. Decide, and — per the template's section 9 — write the decision, why, and one alternative you rejected.

Your call: the component boundaries, the data model shape, and at least one documented trade-off you'll defend on Friday.

Now sanity-check the doc against the spec with your agent:

Prompt to your agent:
"Compare docs/ARCH.md against docs/SPEC.md. Does the architecture
support every v1 feature? Flag anything in the data model or API
that a spec feature needs but the arch is missing. Don't edit —
just report."
git add docs/ARCH.md
git commit -m "ARCH.md v1: components, data model, API, deploy plan"
git push
✅

Keep it current. From here on, if the build drifts from this doc, fix the doc. A stale architecture doc is worse than none — it lies with confidence.

✅ Definition of done

  • ✓ docs/ARCH.md has a big-picture summary and a diagram.
  • ✓ A components table maps each part to a responsibility, tech and where it runs.
  • ✓ Your data model turns the spec's "things" into tables with columns and relationships.
  • ✓ Access rules are written in words; a small API contract exists (or a clear "no backend in v1").
  • ✓ At least one decision + trade-off is documented.
  • ✓ ARCH.md v1 is committed and pushed, and it supports every v1 feature in your spec.

🎤 Friday demo — 5 minutes

  1. Show your diagram and narrate the flow from browser to data.
  2. Walk one table and its access rule — who can see this row?
  3. Explain what your backend is for (or why you don't need one yet).
  4. Your one documented decision — and the alternative you rejected.
  5. One thing the agent's spec-vs-arch check flagged.

Stuck? Common snags

Do I actually need a backend?

Not for everything. Supabase handles auth and basic reads/writes directly from the front end. A backend earns its place when logic shouldn't live in the browser — computing streaks, weekly summaries, or anything that must be trusted. It's fine for your v1 arch to start front-end-only and add the backend in week 7.

My data model feels over-engineered

Model only what your v1 features need — no "just in case" tables. If a column doesn't back a feature in SPEC.md, cut it. You can always migrate later; an app you can explain beats a schema you can't.

I don't know Postgres types yet

That's fine — approximate now (text, int, date, uuid, timestamptz cover most cases) and let the agent refine them when you build the tables in week 6. The shape and ownership of the data matter more than exact types today.
